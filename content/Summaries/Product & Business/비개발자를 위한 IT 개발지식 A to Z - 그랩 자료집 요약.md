# 비개발자를 위한 IT 개발지식 A to Z - 그랩 자료집 요약

원문 노트: [[IT 개발자와 일할 때 필요한 모든 개발지식 A to Z 자료 모음집 By 그랩 - 원문]]

## 한 줄 요약

그랩의 Notion 자료집은 기획자·디자이너·PM·PO·스타트업 대표·신입 개발자가 개발자와 협업하기 위해 알아야 할 운영체제, 네트워크, 프론트엔드, 모바일, 백엔드, 클라우드, 데이터, 배포, Git, 분석 도구, 개발 프로세스를 큰 흐름 중심으로 연결해 둔 IT 개발지식 로드맵이다.

## 핵심 메시지

- **개발 지식은 따로따로 외우는 것이 아니라 연결해서 이해해야 한다.** 운영체제, 네트워크, 프론트엔드, 백엔드, 데이터, 클라우드, 배포는 실제 제품 개발 안에서 이어진다.
- **비개발자에게 필요한 것은 구현 세부사항보다 전체 흐름이다.** 개발자가 무엇을 왜 하는지, 어떤 제약과 순서로 일이 진행되는지 이해하는 것이 협업에 더 중요하다.
- **IT 서비스는 프론트엔드·모바일·백엔드·데이터·클라우드가 함께 작동하는 시스템이다.** 사용자가 보는 화면만이 아니라 서버, API, 로그, 데이터 파이프라인, 배포까지 알아야 제품 논의를 더 잘할 수 있다.
- **제품 개발 프로세스를 이해하면 협업 비용이 줄어든다.** 비즈니스 로직, 데이터 트래킹, 테스트, 코드 통합, 코드 리뷰, QA, 배포, DevOps 흐름을 알면 개발자와의 커뮤니케이션이 명확해진다.
- **PM/디자이너/PO에게 개발지식은 의사결정 언어다.** 기술을 직접 구현하지 않더라도 기술적 제약, 비용, 리스크, 우선순위를 이해하는 데 필요하다.

## 자료집 구조

### 1. 개발 배경지식

기초 개발 개념을 빠르게 훑는 파트다.

- 운영체제 & 하드웨어
- 프로세스 & 프로그래밍 언어
- 네트워크
- 라이브러리 & 프레임워크
- API
- Git & Github

이 파트는 개발자와 대화할 때 자주 등장하는 기본 단어들을 이해하기 위한 기반이다.

### 2. 실제 IT 서비스에 사용되는 기술

실제 서비스가 어떤 기술 구성으로 작동하는지 설명하는 파트다.

- 프론트엔드
- 모바일
- 백엔드
- 클라우드
- 클라이언트 로그
- 서버 로그

제품 경험을 볼 때 “화면”만 보지 않고, 화면 뒤의 데이터 흐름과 서버·클라우드 구조까지 생각하게 만드는 구간이다.

### 3. 프론트엔드 웹 큰그림

웹 서비스의 작동 원리와 프론트엔드 개발의 핵심 개념을 다룬다.

- 브라우저에서 웹이 동작하는 과정
- 웹과 API 서버의 통신
- 웹 라이브러리 & 프레임워크
- 반응형 웹
- SPA
- CSR & SSR
- 크로스 브라우징
- HTTP & HTTPS

PM이나 디자이너 입장에서는 화면 요구사항이 실제 브라우저·API·렌더링 방식과 어떻게 연결되는지 이해하는 데 유용하다.

### 4. 모바일 앱 큰그림

앱 서비스의 구조와 앱 유형을 이해하는 파트다.

- 모바일 앱 기본 지식
- IT 서비스에서 앱이 동작하는 과정
- 웹뷰
- 네이티브 앱
- 웹 앱
- 하이브리드 앱
- 크로스 플랫폼 앱
- 웹과 앱의 차이점

앱 기능을 논의할 때 네이티브 구현, 웹뷰, 하이브리드, 크로스 플랫폼의 차이를 이해하면 일정·성능·유지보수 논의가 쉬워진다.

### 5. 백엔드, 서버, 클라우드 큰그림

사용자 화면 뒤에서 서비스가 작동하는 기반을 다룬다.

- 백엔드의 정의
- 데이터베이스
- API 서버
- 파일 스토리지
- 클라우드
- 서버 스케일링
- 모니터링

백엔드는 단순히 “서버 개발”이 아니라 데이터 저장, API 응답, 파일 관리, 클라우드 운영, 장애 대응까지 포함하는 영역이다.

### 6. 프론트엔드와 백엔드 개발자가 회사에서 하는 일

개발자 직군별 업무를 이해하는 파트다.

- 프론트엔드 개발자
  - 프로덕트 디자이너와 협업
  - 프론트엔드 개발자의 역할
- 백엔드 개발자
  - API 개발
  - 데이터베이스 관리
  - 서버 & 클라우드 관리

비개발자가 개발자에게 요청할 때 누구에게 어떤 종류의 질문을 해야 하는지 판단하는 데 도움이 된다.

### 7. 데이터 직군의 데이터 관리

IT 회사에서 데이터가 생성되고 활용되는 흐름을 다룬다.

- 데이터 생성
- 데이터 수집
- 데이터 가공 후 저장, ETL
- 데이터 시각화
- 데이터 분석
- 데이터 엔지니어의 데이터 파이프라인 구축
- 데이터 사이언티스트의 모델 생성·개선·분석

PM/UX 관점에서는 데이터 트래킹 요구사항, 이벤트 설계, 분석 가능한 구조를 이해하는 데 연결된다.

### 8. IT 제품 개발 프로세스

실제 제품 개발이 어떤 순서로 진행되는지 다룬다.

- 제품팀 구성
- 비즈니스 로직 개발
- 데이터 트래킹
- 테스트 작성
- 개발 코드 통합
- 빌드 & 테스트 실행
- 코드 리뷰
- 코드 충돌 해결
- QA
- 서비스 배포
- CD, Continuous Development
- DevOps

비개발자가 제품 출시 일정을 이해하거나 개발 리스크를 논의할 때 핵심이 되는 파트다.

### 9. 분석·그로스 툴

성장하는 IT 회사에서 자주 쓰는 분석 도구를 소개한다.

- 행동 분석 툴: Google Analytics, Amplitude, Hotjar, Google Tag Manager
- A/B 테스트 툴: Google Optimize, Optimizely
- 랜딩페이지 제작 툴: Unbounce, Instapage

제품 개선을 위한 실험과 분석 환경을 이해하는 데 유용하다.

## 관찰과 인사이트

### 비개발자의 개발지식은 “협업 해상도”를 높인다

비개발자가 개발을 깊게 구현할 필요는 없지만, 기술의 전체 흐름을 모르면 요구사항이 모호해지고 커뮤니케이션 비용이 커진다. 예를 들어 “이 버튼 누르면 알림 보내 주세요”라는 요청도 실제로는 프론트 이벤트, API, 백엔드 로직, DB 저장, 푸시 서버, 로그 트래킹, QA까지 연결될 수 있다.

### UX/PM에게 특히 중요한 파트는 API, 로그, 데이터 트래킹, QA다

사용자 경험을 설계하는 사람은 화면 흐름만이 아니라 데이터와 시스템 흐름도 이해해야 한다.

- API를 이해하면 화면과 서버의 책임을 구분할 수 있다.
- 클라이언트 로그와 서버 로그를 이해하면 사용자의 행동과 장애를 추적할 수 있다.
- 데이터 트래킹을 이해하면 제품 지표와 리서치 질문을 연결할 수 있다.
- QA와 배포를 이해하면 출시 전후 리스크를 더 잘 관리할 수 있다.

### 이 자료집은 “순서대로 읽는 로드맵”이자 “필요할 때 찾아보는 사전”이다

페이지 자체가 네비게이터 역할을 한다. 처음 공부할 때는 1장부터 끝까지 읽고, 실무 중 특정 개념이 필요할 때는 링크 모음처럼 찾아보는 방식이 적합하다.

## 실무 활용 예시

### PM/PO가 개발자와 기능을 논의할 때

- 이 기능은 프론트엔드 변경인가, 백엔드/API 변경인가?
- 새로운 데이터 저장이 필요한가?
- 어떤 이벤트를 트래킹해야 하는가?
- 클라이언트 로그와 서버 로그 중 어디서 확인할 수 있는가?
- QA에서 반드시 검증해야 할 케이스는 무엇인가?
- 배포 후 모니터링할 지표는 무엇인가?

### UX 디자이너가 개발 가능성을 확인할 때

- 이 인터랙션은 브라우저/앱 환경에서 구현 가능한가?
- 웹뷰, 네이티브, 하이브리드 구조에 따라 제약이 있는가?
- 반응형 웹이나 크로스 브라우징 이슈가 있는가?
- API 응답 속도나 서버 상태가 경험에 영향을 주는가?
- 디자인 변경이 데이터 트래킹이나 QA 범위에 영향을 주는가?

### 신입/비개발자가 공부할 때 추천 순서

1. 개발 배경지식: 운영체제, 네트워크, API, Git
2. 실제 IT 서비스 기술 구조: 프론트엔드, 모바일, 백엔드, 클라우드
3. 제품 개발 프로세스: 테스트, 코드 리뷰, QA, 배포
4. 데이터와 로그: 트래킹, ETL, 분석, 시각화
5. 분석 도구와 그로스 툴: GA, Amplitude, Hotjar, GTM, A/B 테스트

## 체크리스트

개발자와 협업하기 전 확인할 질문:

- [ ] 요구사항이 화면 변경인지, API 변경인지, DB 변경인지 구분했는가?
- [ ] 사용자 행동을 측정할 이벤트와 로그가 정의되어 있는가?
- [ ] 프론트엔드와 백엔드의 책임 범위가 정리되어 있는가?
- [ ] 모바일 앱/웹/웹뷰/네이티브의 제약을 고려했는가?
- [ ] 테스트와 QA에서 확인해야 할 케이스를 적었는가?
- [ ] 배포 후 모니터링할 지표나 로그를 정했는가?
- [ ] 개발자가 이해할 수 있는 용어로 요구사항을 설명했는가?

## 내 작업에 적용할 포인트

- UX 리서치나 제품 기획 노트에서 “개발 영향 범위” 섹션을 추가한다.
- 기능 아이디어를 정리할 때 프론트엔드/API/DB/로그/QA/배포 관점을 함께 적는다.
- 리서치 인사이트가 제품 구현으로 이어질 때 필요한 기술 협업 질문을 체크리스트화한다.
- 비개발자 대상 문서에서는 구현 세부 코드보다 시스템 흐름과 용어 관계를 먼저 설명한다.
