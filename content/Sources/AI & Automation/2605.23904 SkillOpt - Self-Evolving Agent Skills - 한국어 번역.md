## 초록 번역

오늘날 에이전트 스킬은 사람이 직접 작성하거나, LLM이 한 번에 생성하거나, 느슨하게 통제되는 자기 수정 과정을 통해 진화한다. 하지만 이들 중 어느 것도 스킬에 대한 딥러닝 옵티마이저처럼 동작하지 않으며, 피드백을 받아도 시작점보다 안정적으로 개선된다고 보장하지 않는다. 저자들은 스킬을 고정된 에이전트의 외부 상태로 보고, weight-space optimization을 재현 가능하게 만드는 것과 같은 규율로 훈련해야 한다고 주장한다.

**SkillOpt**는 저자들이 아는 한, 에이전트 스킬을 위한 최초의 체계적이고 통제 가능한 text-space optimizer다. 별도의 optimizer model이 점수화된 rollout을 단일 skill document에 대한 제한된 add/delete/replace edit으로 바꾸고, 해당 edit은 held-out validation score를 엄격히 개선할 때에만 수락된다. textual learning-rate budget, rejected-edit buffer, epoch-wise slow/meta update는 스킬 훈련을 안정화하며, 배포 시에는 추가 모델 호출을 전혀 늘리지 않는다.

6개 benchmark, 7개 target model, 3개 execution harness(direct chat, Codex, Claude Code)에서 SkillOpt는 평가된 52개 (model, benchmark, harness) cell 전체에서 최고 또는 공동 최고였고, human skill, one-shot LLM skill, Trace2Skill, TextGrad, GEPA, EvoSkill을 모두 이겼다. GPT-5.5에서는 no-skill accuracy 대비 direct chat에서 +23.5 point, Codex agentic loop에서 +24.8 point, Claude Code 안에서 +19.1 point를 끌어올렸다. Transfer experiment는 최적화된 skill artifact가 model scale, Codex/Claude Code 실행 환경, 인접 math benchmark로 이동해도 추가 최적화 없이 가치를 유지한다는 점도 보여준다.

## 1. Introduction 번역

Frontier language model은 단일 프롬프트 호출기에서부터 tools, files, verifier를 사용하는 multi-step execution harness에 이르기까지 점점 더 agent로 배포되고 있다. 이런 환경에서 domain adaptation은 더 이상 model weight나 prompt만의 문제가 아니다. 에이전트가 증거를 수집하고, 도구를 호출하고, domain convention을 따르고, output을 formatting하는 **절차**를 개선해야 한다.

Agent skill은 이 procedural adaptation을 위한 자연스러운 인터페이스다. skill은 procedure, domain heuristic, tool policy, output constraint, failure mode를 포장하는 portable natural-language artifact이며, frozen agent가 외부 텍스트를 통해 적응할 수 있게 한다.

반복적으로 적응해야 하는 대상이 에이전트의 절차라면, skill document 자체도 trainable해야 한다. 그러나 closed frontier model에서는 weight adaptation이 불가능한 경우가 많고, open model에서도 비용이 크다. 사람이 쓴 skill이나 one-shot skill은 target domain 또는 harness 아래에서 brittle하다. 기존 시스템들은 execution experience를 reusable textual artifact로 바꾸거나 trajectory lesson을 증류하고, failure analysis로 skill folder를 개선하고, domain-specific skill library를 만들거나 trajectory feedback으로 prompt를 최적화해왔다. 하지만 더 기본적인 질문은 남아 있다.

> skill이 adaptation layer라면, skill은 어떻게 최적화되어야 하는가?

이 논문의 핵심 아이디어는 skill editing을 **통제 가능한 domain-adaptation process**로 취급하는 것이다. skill document는 외부 상태, 추가 frontier model은 optimizer, 그리고 evidence, step size, validation, update direction은 training-style control로 관리된다.

저자들은 **SkillOpt**를 제안한다. SkillOpt는 target domain, initial skill, adapted target model을 입력으로 받아 다음을 반복한다.

1. trajectory batch를 sample한다.
2. success와 failure를 분석한다.
3. frontier optimizer model에게 구조화된 add/delete/replace edit을 제안하게 한다.
4. textual learning-rate budget 아래 candidate edit을 aggregate/rank한다.
5. bounded update를 skill document에 적용한다.
6. held-out selection split에서 candidate skill을 평가한 뒤, 개선될 때만 수락한다.
7. rejected edit은 negative feedback으로 보존한다.
8. epoch-wise slow/meta update로 장기 regularity를 유지한다.

배포되는 결과물은 약 300~2,000 token 규모의 compact한 `best_skill.md` 파일이며, adapted model과 execution harness는 그대로 유지된다.

이 딥러닝 비유는 장식이 아니라 운영적이다.

- rollout/reflection batch size는 각 edit에 쓰이는 evidence의 noise를 조절한다.
- textual learning rate와 schedule은 skill version이 이전 버전에서 얼마나 멀리 이동할 수 있는지 조절한다.
- held-out gate는 validation 역할을 한다.
- epoch-wise slow/meta update는 momentum처럼 epoch 간 안정적인 방향을 운반한다.

이 안정성이 중요하다. consecutive skill revision이 너무 멀리 또는 일관성 없이 움직이면 rejected edit과 이전 accepted edit이 의미 있는 optimization history를 제공하지 못한다. bounded, validation-gated update를 사용하면 각 revision은 이전 버전과 충분히 가까이 남아 이후 optimizer call이 무엇이 도움이 되었고, 무엇이 실패했고, 무엇을 보존해야 하는지 배울 수 있다.

저자들은 SkillOpt를 6개 benchmark, 7개 target model, 3개 execution mode에서 평가했다. 52개 cell 모두에서 SkillOpt가 최고 또는 공동 최고였다. GPT-5.5 direct chat에서는 SearchQA 77.7→87.3, SpreadsheetBench 41.8→80.7, OfficeQA 33.1→72.1, DocVQA 78.8→91.2, LiveMathematicianBench 37.6→66.9, ALFWorld 83.6→95.5로 향상되었다. 평균적으로 no-skill 대비 +23.5 point 개선이다.

또한 learned artifact는 정확히 훈련한 setting 밖으로도 transfer된다. GPT-5.4에서 학습한 SpreadsheetBench skill은 더 작은 GPT variant도 개선했고, Codex에서 훈련한 spreadsheet skill은 Claude Code로 옮겼을 때 +59.7 point gain을 보였다. OlympiadBench skill은 Omni-MATH에서도 positive gain을 냈다. 이는 skill을 한 번 최적화하고, 텍스트로 audit한 뒤, model weight를 바꾸지 않고도 관련 model/harness/task에 재사용할 수 있음을 보여준다.

## 2. Related Work 번역 요약

### Prompt auto tuning과 agent configuration search

GEPA는 trajectory feedback이 reflective prompt evolution을 이끌 수 있고, 여러 language-agent task에서 reinforcement learning보다 나은 성능을 낼 수 있음을 보였다. ABSTRAL과 EvoTest는 이 아이디어를 단일 prompt에서 multi-agent design document와 test-time agentic system evolution으로 확장했다. 하지만 이들은 주로 prompt, system design, full configuration을 대상으로 하며, 재사용 가능한 domain adaptation skill을 훈련·검증·export하는 데 초점을 맞추지는 않는다.

SkillOpt는 persistent skill document를 최적화한다. 이 문서는 adapted model과 함께 train, validate, export, reuse될 수 있다.

### Skill construction과 skill evolution

SkillsBench와 agentic skills에 관한 SoK는 skill을 tool policy, applicability condition, execution routine, supporting resource를 포함하는 reusable procedural knowledge로 본다. 기존 연구는 lifelong experience, trajectory lesson, skill knowledge base, heterogeneous domain resource로 skill을 만들고, failure analysis, creation-evaluation-revision loop, generator/verifier co-evolution, collective update, reinforcement learning 등을 통해 개선한다.

SkillOpt는 더 좁은 문제를 다룬다. 즉 하나의 compact domain skill을 trajectory batch, reflection minibatch, textual learning rate, validation gate, rejected-edit buffer, slow/meta update 같은 딥러닝식 control로 어떻게 훈련할 것인가다.

## 3. Method 번역

### Problem Setup

Skill `s`는 execution 전에 agent context에 삽입되는 natural-language policy다. Direct-chat benchmark에서는 system 또는 developer instruction 앞에 붙고, tool-use harness에서는 persistent procedural memory가 된다. Target model `M`은 frozen 상태이며, skill optimization을 통해 행동만 적응한다.

SkillOpt는 train, selection, test split을 사용한다.

- train split: experience 생성
- selection split: update gate
- test split: final reporting에만 사용

Optimizer state에는 current skill, best validation-gated skill, cached skill hash, epoch-local rejected-step buffer, optional slow/meta-update state가 들어 있다. 최종적으로 best accepted skill만 `best_skill.md`로 export된다.

### Forward Pass: Rollout Evidence

각 optimization step에서 target model은 current skill을 사용해 train split의 rollout batch를 실행한다. Harness는 task metadata, messages, tool calls, observations, command outputs, final answers, verifier feedback, benchmark-specific context를 기록한다. 이 batch가 evidence unit이다.

작은 batch는 빠르지만 noisy하고, 큰 batch는 skill이 바뀌기 전에 반복되는 pattern을 더 잘 드러낸다. 구현은 여러 rollout batch를 별도로 reflect하고 하나의 update로 merge하는 accumulation도 지원한다.

### Backward Pass: Minibatch Reflection

Optimizer model은 trajectory를 skill edit으로 바꾼다. 먼저 failure와 success를 분리하고, 각 group을 reflection minibatch로 나눈다. 단일 trajectory는 anecdotal fix를 만들기 쉽지만, minibatch는 반복되는 procedural error를 드러낸다.

예시:

- agent가 계속 잘못된 source를 검색한다.
- 답변을 잘못된 format으로 쓴다.
- tool result를 검증하지 않는다.

Failure minibatch는 누락되었거나 교정해야 할 rule을 제안하고, success minibatch는 이미 작동하는 행동을 보존한다. 각 reflection은 structured add/delete/replace edit 또는 rewrite suggestion을 반환한다.

### Bounded Text Updates

SkillOpt의 learning-rate analogue는 edit budget `L_t`다. 이는 step `t`에서 적용할 수 있는 최대 skill edit 수다. Optimizer는 merged edit pool을 expected utility로 rank하고 top `L_t`만 적용한다.

이 점이 ad hoc prompt rewriting과 다르다. Unbounded rewrite는 유용한 rule을 지우거나, incompatible instruction을 넣거나, local failure에 overfit할 수 있다. Bounded update는 continuity를 보존하면서 skill이 새로운 procedure를 획득하게 한다.

SkillOpt는 constant, linear, cosine, autonomous schedule을 지원한다. 기본 cosine schedule은 초기에 더 큰 edit을 허용하고, 뒤로 갈수록 작은 consolidation step으로 줄인다.

### Validation Gate와 Rejected-Edit Buffer

모든 candidate skill은 같은 frozen target model과 harness로 selection split에서 평가된다. current selection score보다 엄격히 개선되면 new current skill이 된다. best score도 넘으면 `best_skill.md`가 된다. 그렇지 않으면 reject된다.

이 gate는 reflection을 unconditional self-editing이 아니라 propose-and-test optimization으로 바꾼다. plausible한 textual diagnosis도 실제 target model 성능을 해칠 수 있기 때문에 중요하다.

Rejected update도 버리지 않는다. Optimizer는 rejected edit과 score drop을 epoch-local buffer에 기록한다. 이후 같은 epoch의 reflection call은 이 buffer를 받아, 실패한 edit을 반복하지 않고 unresolved failure에 집중한다. 배포 시 inference-time cost를 추가하지 않으면서 training 중 negative feedback을 제공하는 장치다.

### Epoch-wise Slow/Meta Update

Fast local edit과 별도로, SkillOpt는 epoch 단위로 더 느린 update를 수행한다. 이는 장기적으로 안정적인 교훈을 skill 안의 protected region이나 optimizer-side meta skill에 보존한다. Step-level edit은 이 protected field를 덮어쓸 수 없다. 따라서 빠른 intra-epoch edit과 느린 cross-epoch consolidation이 분리된다.

### Harness-Agnostic Deployment

SkillOpt의 결과물은 특정 optimizer model 호출을 요구하지 않는 정적 `best_skill.md`다. Direct chat에서는 instruction으로, Codex/Claude Code 같은 tool-use harness에서는 persistent procedural memory로 사용할 수 있다. 중요한 점은 배포 시 model weight update도, 추가 optimizer call도 필요 없다는 것이다.

## 4. Experiments 번역

### 설정

저자들은 QA, spreadsheet, document, math, embodied decision making을 포괄하는 6개 benchmark에서 평가했다.

- SearchQA
- SpreadsheetBench
- OfficeQA
- DocVQA
- LiveMathematicianBench
- ALFWorld

Target model은 frontier-scale GPT부터 smaller Qwen까지 7개였고, execution mode는 direct chat, Codex harness, Claude Code harness였다. Baseline은 no skill, human skill, one-shot LLM skill, Trace2Skill, TextGrad, GEPA, EvoSkill이다.

### Main Results

Table 1의 핵심 결과는 강하다. 52개 (target model, benchmark, harness) cell 각각을 비교 단위로 보고, no-skill, human-skill, LLM-skill, Trace2Skill, TextGrad, GEPA, EvoSkill 중 가장 강한 per-cell baseline과 비교했을 때, **SkillOpt는 52/52 cell에서 최고 또는 공동 최고**였다.

GPT-5.5 direct chat에서 여섯 benchmark 평균은 no skill 58.8에서 SkillOpt 82.3으로 상승했다. 이는 +23.5 point 절대 개선이다. per-cell best baseline을 oracle처럼 고른 평균은 76.9였으므로, SkillOpt는 이 oracle baseline보다도 +5.4 point 높았다.

Benchmark별 no-skill 대비 개선폭은 다음과 같다.

- SearchQA: 77.7 → 87.3 (+9.6)
- SpreadsheetBench: 41.8 → 80.7 (+38.9)
- OfficeQA: 33.1 → 72.1 (+39.0)
- DocVQA: 78.8 → 91.2 (+12.4)
- LiveMathematicianBench: 37.6 → 66.9 (+29.3)
- ALFWorld: 83.6 → 95.5 (+11.9)

절차가 중요한 benchmark에서 개선폭이 가장 컸다. SpreadsheetBench, OfficeQA, LiveMathematicianBench는 tool use, format, verification, domain convention이 중요하기 때문에 zero-shot frontier model의 한계가 잘 드러난다.

개선은 frontier-scale target에만 국한되지 않았다. 6개 benchmark 평균으로 SkillOpt는 GPT-5.4를 +12.7 point, GPT-5.4-mini를 +15.4, GPT-5.4-nano를 +26.7, GPT-5.2를 +16.6, Qwen3.5-4B를 +19.2, Qwen3.6-35B-A3B를 +9.1 끌어올렸다. 작은 모델일수록 상대적 이득이 큰 경향이 있었고, 이는 compact skill artifact가 작은 모델의 weight에 아직 없는 procedural knowledge를 공급한다는 해석과 맞다.

Codex harness에서도 GPT-5.5는 5개 benchmark 전체에서 SkillOpt가 최고였고, no skill 대비 평균 +24.8 point, EvoSkill 대비 +14.0 point를 보였다. Claude Code harness에서도 5개 benchmark 전체에서 최고였으며, no skill 대비 +19.1, EvoSkill 대비 +3.2 point였다.

### Alternative Explanations

저자들은 SkillOpt의 효과가 단순히 prompt length 때문이 아니라고 주장한다. Human skill은 이미 145~516 token이고 one-shot LLM skill보다 긴 경우도 많지만, 모든 direct-chat model row에서 SkillOpt에 밀렸다. Learned artifact도 여전히 compact하다.

또한 optimizer capacity만의 효과도 아니다. SkillOpt는 GPT-5.4-nano 같은 작은 target에서도 baseline을 앞섰고, target-matched optimizer도 strong optimizer gain의 상당 부분을 회복했다. Harness result는 SkillOpt가 하나의 skill format만 exploiting한 것이 아니라는 점도 보여준다.

## 5. Ablations 번역

Ablation은 GPT-5.5를 target과 optimizer로 사용해 진행했다. 전체 메시지는 SkillOpt가 충분한 evidence, bounded textual learning rate, rejected-edit feedback, epoch-wise slow/meta update에서 이득을 얻는다는 것이다.

### Evidence와 batch size

Procedural benchmark는 더 많은 training evidence를 보상했다. SpreadsheetBench는 training partition 1%에서 100%로 늘릴 때 47.5→78.0으로 상승했고, LiveMathematicianBench는 59.1→70.5로 상승했다. SearchQA는 headroom이 적어 20% 이후 84~86 근처에서 포화했다.

Reflection minibatch size와 rollout batch size를 바꿔도 결과는 대체로 안정적이었다. 이는 headline gain이 취약한 prompt-search batch size의 산물이 아니라, scoring된 evidence를 기반으로 한 update의 효과임을 시사한다.

### Textual learning rate와 schedule

Edit budget `L_t`를 1, 2, 4, 8, 16으로 sweep했을 때, 작거나 중간 정도의 edit budget이 전반적으로 경쟁력 있었다. Constant, cosine, linear schedule 모두 강한 결과를 보였다. 중요한 것은 특정 scheduler가 아니라 **moderate bounded edit budget**이라는 원리다. Budget 없이 rewrite하는 baseline은 더 낮았다.

### Slow/meta update와 rejected-edit buffer

Rejected-edit buffer를 제거하면 SearchQA, SpreadsheetBench, LiveMath에서 각각 1.6, 4.6, 2.4 point 하락했다. 이는 rejected edit이 default loop의 stabilizer 역할을 한다는 점을 뒷받침한다.

Slow/meta update를 제거하는 ablation은 특히 SpreadsheetBench에서 큰 하락을 보였다. meta skill과 slow update를 모두 제거하면 77.5에서 55.0으로 떨어졌다. 이는 장기 evidence stream과 protected-region contract가 durable procedural lesson을 보존하는 데 중요하다는 뜻이다.

Validation gate는 엄격하다. Candidate skill은 selection split score가 current selection score보다 **strictly greater**할 때만 accept된다. Tie는 reject된다. 따라서 deployed skill은 조용히 drift하지 않는다. 또한 각 step은 `edit_apply_report.json`을 기록해 `best_skill.md`의 모든 변경 출처를 회수 가능하게 한다.

## 6. Analysis and Transfer 번역

### Cross-model transfer

GPT-5.4에서 훈련한 SpreadsheetBench skill은 GPT-5.4-mini와 GPT-5.4-nano로 transfer되어 각각 +9.4, +3.0 gain을 냈다. LiveMath skill도 GPT-5.4-mini와 GPT-5.4-nano에서 +4.5, +5.6 gain을 냈다. 모든 cross-model row는 positive였다.

이는 일부 learned procedure가 target-model agnostic하다는 것을 시사한다.

### Cross-harness transfer

가장 뚜렷한 deployment signal은 harness shift였다. Codex loop에서 훈련한 SpreadsheetBench skill은 Claude Code로 transfer되어 no-skill baseline 22.1을 81.8로 끌어올렸다. +59.7 point gain이다. 반대 방향인 Claude Code → Codex transfer도 27.5→71.1, +43.6 point였다.

두 harness는 tool/file API와 command surface가 다르기 때문에, 이 positive transfer는 learned rule이 단순한 harness-specific command recipe만은 아니라는 점을 보여준다. SpreadsheetBench의 경우 workbook-level procedure, structure-first inspection, formula-aware verification, static-value materialization 같은 규칙이 transfer된 것으로 보인다.

### Cross-benchmark transfer

OlympiadBench에서 학습한 skill은 Omni-MATH에서도 세 가지 model scale 모두에서 positive gain을 냈다. 이 gain은 cross-model이나 cross-harness보다 작지만, test instance와 answer-format convention이 바뀌어도 reusable mathematical procedure가 남아 있음을 시사한다.

### Optimizer strength

SkillOpt의 optimizer는 offline training loop에서만 사용되고 배포 시에는 호출되지 않는다. 따라서 stronger optimizer는 deployment cost를 늘리지 않고 skill 품질을 높일 수 있다. GPT-5.5 strong optimizer는 target-matched optimizer보다 모든 cell에서 더 큰 absolute gain을 냈지만, target-matched optimizer도 strong optimizer gain의 56~74%를 회복했다. 이는 SkillOpt가 단순히 강한 teacher가 약한 student에게 지식을 증류하는 pipeline만은 아니며, optimization loop 자체가 상당한 가치를 만든다는 것을 보여준다.

## 7. Learned Skills 번역

SkillOpt의 중심 전제는 trainable object가 작고 inspectable한 텍스트 문서로 남아야 한다는 것이다. GPT-5.5/GPT-5.5 skill run에서 최종 `best_skill.md`는 379~1,995 token 범위였고, median은 약 920 token이었다. 가장 긴 skill도 현대 frontier model의 system-prompt budget에 비하면 작고, domain practitioner가 몇 분 안에 읽고 audit하고 edit할 수 있는 수준이다.

흥미로운 점은 큰 성능 향상이 매우 적은 accepted edit에서 나왔다는 것이다. 6개 benchmark에서 `best_skill.md`에 실제 commit된 edit 수는 1~4개, median 2.5개였다. LiveMathematicianBench의 +29.3 point gain과 OfficeQA의 +39.0 point gain은 각각 단 하나의 accepted edit에서 나왔다. 이는 validation gate가 많은 제안을 reject하고, 실제 배포 skill을 compact하게 유지한다는 직접적 증거다.

대표 learned rule은 다음과 같다.

- **SearchQA:** clue wording에서 expected answer type을 추론한 뒤, 함께 나타나는 distinctive evidence가 지지하는 가장 짧은 canonical entity를 선택하라.
- **SpreadsheetBench:** workbook structure와 formula를 먼저 검사한 뒤, Excel recalculation에 의존하지 말고 요청된 target range 전체에 evaluated static value를 작성하라.
- **OfficeQA:** oracle parsed page를 primary evidence로 취급하고, table/date/unit context를 lock하며, 추가 label 없이 요청된 rounded value만 출력하라.
- **DocVQA:** table, form, chart, legend에서는 먼저 question을 정확한 visual row/header/field에 bind한 뒤, aligned answer span만 복사하라.
- **LiveMathematicianBench:** strongest-statement MCQ에서는 theorem strength로 choice를 rank하고, true but weaker corollary보다 정당화된 stronger-result option을 선호하라.
- **ALFWorld:** horizon-aware visited/frontier ledger를 유지하고, 같은 유형의 실패가 반복되면 search를 diversify하며, target을 들고 있기 전에는 destination 재방문을 피하라.

이 rule들은 instance-specific하지 않고 procedural하다. 특정 질문, 파일, entity를 이름으로 언급하지 않는다. 대신 answer formatting, evidence binding, search-frontier management처럼 frontier model이 zero-shot으로 일관되게 적용하지 못하는 discipline을 인코딩한다.

## 8. Qualitative Skill Evolution 번역

### ALFWorld

초기 ALFWorld skill은 일반적인 household plan을 제공했다. target object를 찾고, 집어 들고, 필요하면 변형한 뒤, destination에 놓는 방식이다. Accepted edit은 이 plan을 더 stateful하고 loop-prone하지 않게 만들었다. 최적화된 skill은 정확한 object-name matching을 학습해 mug, cup, pan, pot 같은 관련 object를 서로 대체하지 않도록 했다. 또한 visited-location memory를 추가해 이미 확인한 likely location을 반복 방문하기보다 unvisited receptacle과 surface를 우선하도록 했다.

### SpreadsheetBench

SpreadsheetBench의 초기 skill은 workbook을 검사하고 답을 쓰라는 수준이었다. SkillOpt는 여기에 workbook structure-first inspection, formula dependency 확인, recalculation에 의존하지 않는 static value materialization, target range 전체 확인 같은 절차적 제약을 추가했다. 이는 spreadsheet task에서 output format과 workbook state가 정확히 맞아야 하는 문제를 해결한다.

## 결론 번역

SkillOpt는 agent skill을 frozen agent의 외부 상태로 보고, scored rollout, minibatch reflection, bounded add/delete/replace edit, validation gate, rejected-edit buffer, slow/meta update를 통해 훈련하는 text-space optimizer다. 실험 결과는 compact skill document를 통제된 방식으로 최적화하는 것이 model weight를 바꾸지 않고도 다양한 agent benchmark와 harness에서 큰 성능 향상을 만들 수 있음을 보여준다.

가장 중요한 점은 배포 artifact가 여전히 작고 읽을 수 있는 `best_skill.md`라는 것이다. Training cost는 offline에서 한 번 지불되고, 배포 시에는 additional model call 없이 target model과 harness에 skill만 추가된다. Transfer 결과는 이 artifact가 특정 training setting의 overfit prompt가 아니라 reusable procedural knowledge를 담고 있을 가능성을 보여준다.

## Appendix B: Limitations 번역 요약

논문의 한계는 다음과 같이 정리할 수 있다.

- SkillOpt는 training-time rollout과 optimizer call을 필요로 하므로, benchmark나 domain에 따라 비용이 커질 수 있다.
- Validation gate는 selection split 품질에 의존한다. Selection split이 실제 deployment distribution을 대표하지 못하면 잘못된 skill이 선택될 수 있다.
- Skill이 compact하더라도, learned rule이 모든 domain에서 안전하게 transfer된다고 보장할 수 없다.
- Strong optimizer를 사용할수록 더 좋은 skill이 나오는 경향이 있지만, 이는 training-time resource와 access에 영향을 받는다.
- 논문은 benchmark 중심 평가이므로, 실제 production agent workflow에서는 tool permission, privacy, audit, governance, human review가 추가로 필요하다.
