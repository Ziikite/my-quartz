# AI 워크플로우 및 프롬프트 대방출 - 구조화 요약

## 한 줄 요약

Obsidian을 중심 저장소로 두고 Readwise, Paperpile, Apple Notes, Cursor, NotebookLM, ChatGPT/Claude를 단계별로 조합해 `수집 → 정리/요약 → 글쓰기/개선`까지 이어지는 개인 지식관리 PKM 워크플로우를 구축하는 방법을 설명한 글이다.

## 핵심 구조

### 1. Capture / Ingest: 정보 수집

- 목표: 여러 소스에서 들어오는 지식과 경험을 Obsidian으로 모으기
- 주요 입력 소스:
  - 웹사이트
  - 전자책
  - 논문
  - 비디오
  - 미팅 노트
  - 일지와 개인 경험 기록
- 사용 도구:
  - Readwise: 웹 클리핑, 하이라이트, 주석을 Obsidian으로 가져오기
  - Paperpile: 논문 메타데이터, PDF, 참고문헌 관리
  - Apple Notes: 음성 인식 기반 미팅 노트와 아이디어 기록
  - Obsidian: 최종 지식 저장소
- 현재 병목:
  - Apple Notes에서 Obsidian으로 옮기는 과정이 아직 수동이다.
  - Obsidian과 자동 싱크되는 음성 메모 워크플로우가 필요하다.

### 2. Distill / Organize: 정리와 요약

- 목표: 누적되는 노트, 일지, 회의록, 독서 기록을 다시 쓸 수 있는 지식으로 압축하기
- 핵심 아이디어:
  - 인간의 뇌가 경험을 요약해 장기기억으로 저장하듯, AI를 이용해 개인 지식 베이스를 요약·재구성한다.
  - 일간 기록을 주간 리뷰로, 개별 학습을 주제별 지식으로 변환한다.
- 사용 도구별 역할:
  - ChatGPT / Claude
    - 입문자가 가능성을 실험하기 좋다.
    - 단점: 매번 문서를 직접 넣어야 한다.
  - NotebookLM
    - 인용이 포함된 요약에 강하다.
    - 큰 고정 자료를 대상으로 할 때 유용하다.
    - 단점: 소스를 매번 넣는 과정이 번거롭다.
  - Cursor
    - Obsidian Vault를 직접 인덱싱해 지식 베이스 기반 질의·요약을 수행할 수 있다.
    - Cursor Rules와 프롬프트를 조합하면 PKM Copilot처럼 쓸 수 있다.
- 현재 병목:
  - 모델 컨텍스트 제한 때문에 한 번에 너무 많은 문서를 처리하기 어렵다.
  - 해결 전략: 일간 → 주간 → 월간처럼 단계별 요약을 만든다.
  - 여러 문서에 태그를 붙이거나 일괄 수정하는 작업은 LLM API 스크립트가 더 적합할 수 있다.

### 3. Write / Refine: 글쓰기와 개선

- 목표: PKM에 축적된 자료를 바탕으로 더 충실한 글을 쓰기
- 글쓰기 흐름:
  - 내부 자료 조사: Obsidian 안의 기존 노트 활용
  - 외부 자료 조사: 필요시 deep research 수행
  - 아웃라인 작성
  - AI로 아웃라인 확장 및 보완
  - AI로 문단 작성
  - 사람이 최종 검토 후 발행
- 사용 도구:
  - Obsidian: 자료와 초안 관리
  - Cursor: 문서 편집, diff 검토, AI 보조 작성
- 글쓰기용 프롬프트 모듈:
  - Outline Expander, OEX: 아웃라인 확장과 논리 보완
  - Paragraph Writer, PRW: 아웃라인을 문단으로 작성
  - Draft Enhancer, DEN: 초안의 흐름, 균형, 문법, 사실성 개선
  - Devil's Advocate, DAV: 비판적 리뷰와 개선점 제안
  - Korean Translator, KT: 자연스러운 한국어 문체로 번역·윤문
- 현재 병목:
  - 내용은 풍부해지지만 AI 특유의 밋밋한 문체가 남을 수 있다.
  - 개인의 고유한 문체를 AI가 더 잘 반영하도록 writing sample을 활용하는 방법이 과제다.

## Cursor를 PKM Copilot으로 쓰는 방식

### Cursor Rules의 핵심 원칙

- 이 저장소를 코드 저장소가 아니라 개인 지식 베이스로 취급하게 한다.
- 요청하지 않으면 코딩 관련 제안을 하지 않게 한다.
- 가능하면 채팅 답변보다 실제 마크다운 파일을 직접 수정하게 한다.
- 변경 범위는 최소화한다.
- 문서 간 연결은 Obsidian의 `[[위키링크]]` 형식을 사용한다.
- 요청 유형을 먼저 판별하게 한다.

### 요청 유형별 역할

- Answer KB Questions
  - 지식 베이스에서 관련 내용을 찾아주는 사서 역할
  - 가능하면 원본 노트로 이어지는 위키링크 제공
- Reflection Partner
  - Journal 폴더의 최근 경험을 바탕으로 자기성장을 돕는 상담자 역할
- Reading Partner
  - Reading 폴더의 독서 기록을 함께 해석하는 독서 동반자 역할
- Writing Partner
  - Writing 폴더의 아웃라인, 문단, 초안을 함께 개선하는 글쓰기 파트너 역할

## 재사용 가능한 프롬프트 패턴

### Weekly Journal Review

- 입력: 특정 주의 일간 Journal
- 처리:
  - 대상 주간의 모든 일지를 찾는다.
  - 누락되거나 빈 일지도 신호로 해석한다.
  - 가능하면 전주 리뷰와 비교한다.
- 질문:
  - 달성한 목표는 무엇인가?
  - 놓친 목표와 과제는 무엇인가?
  - 하이라이트와 로우라이트는 무엇인가?
  - 이 기간에 무엇을 배웠는가?
  - 다음 주에 무엇을 해야 하는가?
- 출력:
  - 한국어 요약
  - 원문 표현을 살린 인용
  - Obsidian 위키링크로 출처 표시
  - `Journal/Weekly Review`에 저장

### Topic Summary

- 입력:
  - 특정 Topic 폴더 아래의 노트
  - 관련 Reading 노트
  - 관련 Journal 노트
- 처리:
  - 폴더 전체와 하위 토픽을 훑는다.
  - 모든 글의 핵심 포인터를 정리한다.
- 출력:
  - 너무 자세한 요약보다 각 글로 연결되는 지도 형태의 요약
  - 위키링크로 출처 표시

### Outline Expander, OEX

- 목적: 글을 쓰기 전 아웃라인을 더 탄탄하게 만들기
- 수행:
  - 논리적 빈틈 채우기
  - 흐름을 자연스럽게 재구성하기
  - 관련 노트를 위키링크로 연결하기
  - 개선 아이디어 제안하기

### Paragraph Writer, PRW

- 목적: 아웃라인을 실제 문단으로 확장하기
- 수행:
  - 아웃라인의 표현을 최대한 살리기
  - 새 내용을 추가했다면 이유 설명하기
  - 관련 KB 내용을 끌어오기
  - 기존 아웃라인을 대체하지 말고 아래에 문단을 추가하기

### Draft Enhancer, DEN

- 목적: 초안을 과하게 바꾸지 않으면서 품질을 높이기
- 수행:
  - 전체 흐름과 전환 확인
  - 섹션·문단 길이 균형 조정
  - 명백한 문법 오류와 의심스러운 사실 수정
  - 리뷰 비용을 줄이기 위해 변화량은 최소화

### Devil's Advocate, DAV

- 목적: 글을 비판적으로 검토하기
- 수행:
  - 핵심 주장 비판
  - 대안적 관점 제시
  - 사실과 문법 점검
  - 개선 제안

### Korean Translator, KT

- 목적: 번역투를 줄이고 자연스러운 한국어 글로 다듬기
- 수행:
  - 어색한 영한 번역체 수정
  - 개인적이고 자연스러운 어조 유지
  - 어미 반복을 줄여 리듬 만들기
  - 필요시 한국 독자에게 어색한 내용 지적

## 주요 관찰 포인트

### 1. Obsidian은 저장소, Cursor는 작업 인터페이스

이 워크플로우에서 Obsidian은 지식을 보관하고 연결하는 장소이고, Cursor는 그 지식을 읽고 고치고 재구성하는 작업 인터페이스에 가깝다. 핵심은 “AI 앱 하나를 잘 쓰는 것”이 아니라 “내 노트 전체를 AI가 다룰 수 있는 저장소로 만드는 것”이다.

### 2. AI 활용의 핵심은 단계화다

한 번에 모든 노트를 처리하려 하면 컨텍스트 제한과 품질 저하가 생긴다. 그래서 일간 → 주간 → 월간, 개별 노트 → 토픽 요약 → 글쓰기처럼 중간 산출물을 만드는 방식이 중요하다.

### 3. 프롬프트보다 역할과 파일 구조가 더 중요하다

좋은 프롬프트도 중요하지만, AI가 어떤 폴더를 어떤 의미로 해석해야 하는지 알려주는 규칙이 더 큰 영향을 준다. Journal, Reading, Topics, Writing 같은 폴더 구조와 요청 유형 정의가 AI의 품질을 안정화한다.

### 4. 자동화와 수동 검토의 균형이 필요하다

수집과 1차 요약은 자동화할수록 좋지만, 최종 글쓰기와 발행은 사람이 검토해야 한다. 특히 문체, 관점, 과장된 주장, 사실 오류는 여전히 사람이 잡아야 한다.

### 5. 개인 지식관리에서 “요약”은 단순 압축이 아니다

요약은 정보를 줄이는 행위가 아니라, 나중에 다시 생각하고 쓸 수 있도록 구조를 재배치하는 과정이다. AI를 쓰면 단순 정리보다 “다음 행동으로 이어지는 지식”을 만들기 쉽다.

### 6. AI 문체 문제는 별도 설계가 필요하다

내용 충실도는 올라가지만 문체가 평준화될 수 있다. 개인의 문체 샘플, 선호 표현, 금지 표현, 독자층을 별도 규칙으로 관리하면 개선 여지가 크다.

## 활용 예시

### 예시 1. 내 주간 회고 자동화하기

- 매일 Obsidian에 짧은 일지를 남긴다.
- 주말에 Cursor에게 이번 주 Journal을 읽고 Weekly Review를 만들게 한다.
- 결과물에는 달성한 목표, 놓친 일, 감정 패턴, 다음 주 과제를 포함한다.
- 매월 말에는 주간 리뷰 4개를 다시 요약해 Monthly Review를 만든다.

### 예시 2. 독서 노트를 주제별 지식으로 바꾸기

- Readwise로 책과 아티클 하이라이트를 Obsidian에 모은다.
- 특정 주제 폴더를 만들고 관련 독서 노트를 연결한다.
- Cursor에게 Topic Summary를 요청해 핵심 주장, 반복되는 아이디어, 반대 관점을 정리하게 한다.
- 요약 페이지를 글쓰기나 프로젝트 기획의 출발점으로 사용한다.

### 예시 3. 글쓰기 워크플로우 만들기

- Obsidian에서 글 주제와 관련된 노트를 검색한다.
- 먼저 아웃라인을 직접 만든다.
- OEX로 빠진 논리와 연결 가능한 노트를 보완한다.
- PRW로 각 아웃라인을 문단으로 확장한다.
- DEN으로 흐름과 균형을 다듬는다.
- DAV로 약한 주장과 반론을 찾는다.
- 최종 문체와 관점은 직접 수정한다.

### 예시 4. 회의록을 실행 가능한 지식으로 바꾸기

- Apple Notes나 음성 메모로 회의 내용을 빠르게 기록한다.
- Obsidian으로 옮긴 뒤 AI에게 다음 항목으로 재구성하게 한다.
  - 결정사항
  - 액션 아이템
  - 후속 질문
  - 관련 프로젝트 링크
  - 다음 회의 전 준비할 것
- 이후 주간 리뷰에서 회의 액션 아이템이 실제로 처리됐는지 확인한다.

### 예시 5. 나만의 Cursor Rules 만들기

내 Obsidian Vault에 맞게 아래 요소를 규칙으로 정의하면 좋다.

- 폴더 의미:
  - Journal: 일상 기록과 회고
  - Reading: 독서·아티클·논문 기록
  - Topics: 주제별 지식 정리
  - Projects: 실행 중인 프로젝트
  - Writing: 글 초안과 발행물
- 응답 원칙:
  - 가능한 경우 원본 노트 링크 제공
  - 새 글을 만들 때는 출처 노트 링크 유지
  - 이미 있는 문단을 고칠 때는 최소 수정
  - 확신이 낮은 내용은 추측이라고 표시
  - 개인 문체를 유지하고 과한 AI식 표현 줄이기

## 내 워크플로우에 적용할 때의 체크리스트

- [ ] Obsidian 안의 폴더 구조가 AI가 이해하기 쉽게 나뉘어 있는가?
- [ ] Journal, Reading, Topics, Writing의 역할이 명확한가?
- [ ] Readwise나 다른 클리핑 도구로 수집이 자동화되어 있는가?
- [ ] 매일/매주/매월 리뷰 템플릿이 있는가?
- [ ] Cursor Rules 또는 유사한 시스템 프롬프트가 있는가?
- [ ] 글쓰기용 OEX, PRW, DEN, DAV 같은 단축 프롬프트가 있는가?
- [ ] AI가 만든 결과물을 사람이 검토하는 단계가 있는가?
- [ ] 내 문체 샘플과 선호 표현이 따로 정리되어 있는가?

## 한계와 주의점

- AI 요약에 의존하면 직접 읽고 생각하는 시간이 줄어들 수 있다.
- 개인 일지와 회의록을 AI 도구가 인덱싱할 때 프라이버시 리스크를 점검해야 한다.
- Cursor의 검색·참조 기능이 항상 완전한 RAG 품질을 보장하는 것은 아니다.
- 자동화가 많아질수록 시스템 유지 비용도 커진다.
- 지식관리 시스템은 생산성을 높이는 도구이지, 그 자체가 목적이 되면 안 된다.

## 다음 액션 아이디어

- 내 Obsidian Vault에 `Journal`, `Reading`, `Topics`, `Projects`, `Writing` 폴더를 만들거나 정리한다.
- 주간 리뷰 템플릿을 먼저 만든다.
- Cursor 또는 다른 AI 편집기에 Vault를 열고, 개인용 Rules를 작성한다.
- 가장 반복적인 작업 하나만 골라 자동화한다. 예: “이번 주 일지 요약”, “읽은 글 요약”, “회의록 액션 아이템 추출”.
- 2주 정도 사용한 뒤, 실제로 시간이 줄었는지와 생각의 질이 좋아졌는지 회고한다.
